SYNOPSIS

DESCRIPTION

A gofer type is a collection of recipes for computing a particular kind of output value, so called because it goes and gets the output value on demand. A value of a gofer type specifies a particular recipe and the ingredients it needs to compute the desired output.

This man page originally described the interface that one would implement by hand; given the increasing maturity of the gofer(n) infrastucture, there is no little reason to do so: it is better to simply use gofer(n)'s gofer define command. As a result, rather than describing the gofer type interface, as such, this man page now documents the gofer type architecture.

OVERVIEW

The Athena application has many orders and tactics that operate on a list of groups. Typically, the order dialogs allow the user to select the desired groups from a list. However, there are many other ways one might want to select a specific set of groups. For example,

  • All non-empty civilian groups
  • All civilian groups resident in neighborhood n
  • All civilian groups with a mood greater than x
  • All civilian groups that support actor a

These are four recipes that a ::gofer::CIVGROUPS type might allow the user to use, in addition to simply selecting the groups from a list.

The first recipe has no ingredients, i.e., no additional parameters required to compute the desired list of groups.

In the second recipe, the ingredient is the identity of the neighborhood, n, and so on for the third and fourth.

Gofer types can be used as parameter types for orders, and especially for tactics. Consider a tactic that requires a list of civilian groups: by using a gofer type, the analyst can select a dynamic set of groups, a set that might change as the simulation runs, simply by specifying the rule used to identify the groups in the set. For this reason, we refer to a gofer type's recipes as gofer rules, or just rules for short.

A gofer type is implemented as a Tcl ensemble command using the gofer(n) infrastructure. Each gofer type has a name, an identifier in upper case like "CIVGROUPS"; the command will always be in the ::gofer namespace, e.g., ::gofer::CIVGROUPS.

Operations

Gofer types support the following operations on their values.

First, a value can be validated. Validation finds any errors, and if valid puts the value in canonical form. In other words, a gofer type is a validation type like all of the others in Athena and Mars.

Second, a narrative can be computed from a value. The narrative is a human-readable string, usually a noun phrase, that describes the output and how it will be computed, e.g., "all non-empty civilian groups resident in N1".

Third, a value can be evaluated, resulting in an output value of the desired type.

Suppose a tactic has a parameter of a gofer type. The tactic will validate parameter values using the gofer type, as it does for any other parameter. It will include the parameter's narrative in its own narrative, e.g.,

Actor JOE funds $25M of ENI for all non-empty civilian groups in N1

Finally, when the tactic executes it will evaluate the value, and make use of the output.

Gofer Dictionaries

Every gofer type value is a Tcl dictionary with at least two keys: _type, which is the name of the type (e.g., "CIVGROUPS"), and _rule, whose value is the name of the rule to apply. Other keys may be added depending on the rule. For example, values for the four rules given above might be represented like this: 

    _type CIVGROUPS _rule ALL
    _type CIVGROUPS _rule BY_NBHOOD n N1
    _type CIVGROUPS _rule BY_MOODGT x 30.0
    _type CIVGROUPS _rule SUPPORTS  a JOE

These values are called gofer dictionaries, or gdicts. In canonical form,

  • The _type key will be the first key.
  • The _rule key will be the second key.
  • The type and rule names will be in upper case.
  • The remaining keys will be only those related to the selected rule.
  • The values of those keys will be in canonical form for their types.

Required Rules

The set of rules will usually vary significantly from one gofer type to another. However, every gofer type must implement at least the BY_VALUE rule, which simply returns its input raw_value.

After all, the analyst will sometimes want to just specify the value to use, rather than telling the application how to compute it. The BY_VALUE rule is how the gofer type supports this.

Thus, 

    _type typename _rule BY_VALUE raw_value {A B C}

will evaluate (if valid) to "A B C".

Raw Value Conversion

Gofer types are a new construct, and there are many existing orders and tactics that ought to use them. For example, the FUNDENI tactic has a glist parameter whose value is a list of civilian groups. At present, the user must always select an explicit list; and there are existing Athena scenarios that use this tactic. How do we transition the tactic to use a gofer type without breaking existing scenarios?

The answer lies in the canonical form of a gofer type value. A raw value will never begin with the token "_type"; a gofer value always will. The application can check for the presence of "_type"; if absent, it can assume that the value is a raw value and convert it to a gofer value using the BY_VALUE rule. (This is another reason why all types must support the BY_VALUE rule.)

EDITING

Gofer type values can be edited by using the gofer field type in dynaform(n) scripts, or by using the goferfield(n) widget directly in other contexts.

Because gofer type values can be arbitrarily complex, each type must define a dynaform(n) script that will be used to display and edit its values in an appropriate way. The form will be used by itself in a dynabox(n) popup, rather than being embedded within a larger form, so it can use all of the space it needs to.

The dynaform script should include prose that makes it clear what the user is selecting. The first data field in the form should always be a selector called _rule; each rule supported by the type should have a case within the selector. For example: 

    rc "" -width 3in -span 3
    label {
        Enter a rule for selecting a set of civilian groups.
    }

    rc "" -for _rule
    selector _rule {
        case BY_VALUE "By name" {
            cc "  " -for raw_value
            enumlonglist raw_value -dictcmd {::civgroup namedict} \
                -width 30 -height 10 
        }
        case RESIDENT_IN "Resident in" {
            cc "  " -for n
            enumlong n -showkeys yes -dictcmd {::nbhood namedict}
        }
    }

Note that the form is free to use whatever text strings it likes to present the rules to the user; thus, the BY_VALUE rule is presented as "By name", which is friendlier for this particular type.

NOTE: The dynaform script must have a field for each gdict key, in the required order; thus, the first two fields in the dynaform script must be the _type and the _rule selector. However, the gofer(n) infrastructure automatically inserts the _type as an invisible context field, so it is omitted from the example.
