DESCRIPTION

This man page describes a component of the athena(n) package that is responsible for executing and managing simulation orders. The actual work is done by a subclass of the order_flunky(n) class; most of the subcommands are identical, though the Athena class adds a few capabilities.

The component is accessible as a subcommand of the athena(n) and athenadb(n) objects; Public Subcommands are accessible from athena(n) and (usually) from athenadb(n), and Private Subcommands only from athenadb(n).

ORDER STATES

The order(n) order processing scheme defines the notion of an order state, which is a symbolic constant representing some broad state of the application. Different orders can be available in different states.

In athena(n), the order states track the scenario states (e.g., PREP, PAUSED) with a single addition, TACTIC, which is used to mark those orders that can safely be used from the EXECUTIVE tactic.

COMMANDS

Public Subcommands

The adb order command takes the following subcommands, where adb is an instance of athena(n).

adb order available name
Returns 1 if the named order exists and is available in the current order state.
adb order canredo
Returns 1 if there's anything on the redo stack to redo.
adb order canundo
Returns 1 if there's anything on the undo stack to undo.
adb order check name ?parm value...?
Validates a set of order parameters and values for the named order, without actually executing the order.
adb order class name
Returns the full TclOO class name of the named order.
adb order make name args...
Creates and returns an instance of the named order, passing the args to the order's constructor.
adb order monitor ?flag?
Sets and queries the order flunky's monitor flag. When the flag is true, any run-time database changes made by an executing order are monitored by the scenario's sqldocument(n) object and the relevant notifier(n) events are sent. When the flag is false, no such monitoring is done.
adb order names
Returns a list of the names of the orders known to the order flunky, i.e., all orders defined by athena(n).
adb order redotext
Returns a narrative string corresponding to the order or transaction on the top of the redo stack, or "" if none.
adb order reset
Clears the undo and redo stacks. This should be done whenever the order state changes.
adb order redo
Redoes the previously undone order or transaction, provided that canredo is true.
adb order send mode name ?option value...?

Creates an instance of order name, initializing it with the order parameters included on the command line in option/value notation, validates it, and executes with the given mode, which should be one of gui, normal, or private. If the mode is gui or normal, the order will be placed on the undo stack and the redo stack will be cleared; if the mode is private, the undo and redo stacks will not be affected.

If the order is rejected, this command throws REJECT with a detailed human-readable error message.

adb order senddict mode name parmdict

Creates an instance of order name, initializing it with the order parameters included in the parmdict, and executes with the given mode, which is just as for the send command.

If the order is rejected, this command throws REJECT with an error dictionary, as returned by order(n)'s errdict method.

adb order transactions flag
Sets and queries the order flunky's transactions flag. If the flag is enabled, each order is executed within an sqldocument(n) transaction. If the order mutator throws an error, the underlying database will be rolled-back. Note that this doesn't affect orders that mutate in-memory state.
adb order transaction narrative script
Executes the script as an "order transaction". The script is a Tcl script that sends a set of orders intended to be treated as a single atomic unit; the transaction will be undone and redone as a unit. The narrative is used for the transaction's undotext and redotext. Note that this kind of "transaction" is entirely distinct from the SQL transactions controlled by transactions.
adb order undotext
Returns a narrative string corresponding to the order or transaction on the top of the undo stack, or "" if none.
adb order undo
Undoes the previously executed order or transaction, provided that canundo is true, and pushes it onto the redo stack.
adb order validate name
Validates the order name, returning it in canonical form if it is valid and throwing INVALID otherwise.
