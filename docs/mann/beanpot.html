DESCRIPTION

An bean(n) object is a TclOO object that can be automatically checkpointed and restored as part of the program's state, i.e., as part of an Athena scenario. Every bean is defined in the context of a beanpot(n) object; beans in the same pot are saved and restored together. The beanpot(n) is also responsible for creating bean objects, assigning bean IDs (which are unique and persistent within a pot) and for various bean-ID-related queries.

Essentially, the application will create a bean pot for each distinct set of beans in memory (i.e., a pot for each scenario), and then use that pot to manage, save, and restore the set of beans. All beans should be created using beanpot(n)'s new subcommand.

Each bean knows what pot it is in; the pot's instance command is stored in the bean's pot instance variable, which cannot be changed.

Bean IDs

Every instance of bean(n) and its subclasses has a unique numeric ID within its bean pot; beans can be looked up given their IDs. The ID is stored in a predefined instance variable called id, which cannot be changed.

The bean ID is assigned by the bean pot's new method when the bean is first created, and persists across checkpoint and restore.

Detecting Changes

Each beanpot(n) instance is a saveable(i), so that the beans in a pot can be checkpointed and restored with the Athena scenario. Consequently, beanpot(n) needs to know when there are unsaved changes to beans. Bean subclass methods can signal that changes have been made by calling their pot's markchanged method. Further, the bean's set method always sets the changed flag. Thus, it is often useful for bean methods to set instance variables using the set method, like this: 

    my set var value

A method that makes many changes to the subclass's own variables can instead use the normal Tcl set command, and call markchanged when done.

The bean(n) class provides lappend and ldelete methods as well, for modifying list-valued variables with notification. In general, all bean(n) methods that update instance variables notify the containing beanpot automatically.

Checkpoint/Restore

Each beanpot(n) instance is a saveable(i), and thus can be plugged into Athena's checkpoint/restore mechanism. Note that bean IDs are preserved across checkpoint/restore, but bean object names are not.

The beanpot(n)'s checkpoint method doesn't actually return a checkpoint. Rather, it's a trigger for the pot to save its beans in its configured -rdb's beans table. The entries are tagged with the pot's -dbid, which defaults to "pot".

Delete/Undelete

The delete and undelete commands support cascading deletion of beans with undo. To delete a bean, pass its ID to delete, which returns a checkpoint-like delete set. To undo the deletion, pass the delete set to undelete.

CLASS COMMANDS

This module defines the following commands:

beanpot name ?options?

Creates a new beanpot object with the given name and options, within which new beans may be created. The pot's contents may be checkpointed and restored. When the pot is reset or destroyed, all beans contained within it are destroyed as well.

The valid options are as follows:

-rdb rdb

Gives the name of the run-time database (RDB), an scenariodb(n) instance (or equivalent) Checkpoints and restores will be done relative to this RDB. Note that in larger applications, the rdb is often an object that wraps the actual RDB handle and provides other services; it may be used as such by the beans in the pot.

If the -rdb is not set, the bean pot cannot be checkpointed or restored.

This option is read-only after the bean pot is created.

-dbid dbid

The application can checkpoint multiple bean pots to a single -rdb. When a bean pot is checkpointed, its entries are tagged with the dbid, which defaults to "pot". If an application uses multiple bean pots with respect to a single scenario, each pot should be given a -dbid corresponding to its role.

It is an error for two bean pots two have the same -rdb and -dbid in the same application.

This option is read-only after the bean pot is created.

INSTANCE COMMANDS

Each instance of the beanpot(n) type has the following subcommands:

pot changed
Returns 1 if any of the pot's bean data has changed, and 0 otherwise. (This is part of beanpot(n)'s saveable(i) interface.)
pot cget option
Returns the value of the named configuration option. See beanpot for a list of options.
pot checkpoint ?-saved?
Checkpoints the beans in the pot to the configured -rdb, tagging them with the configured -dbid. If -saved is given, then the changed flag is cleared. (This is part of beanpot(n)'s saveable(i) interface.)
pot configure option value ...
In principle, this sets the value of one or more of the beanpot(n) configuration options; however, all of the options are read-only after the pot is created.
pot delete id
Deletes the bean with the given id, and returns a "delete set" that can be used to undelete it (under normal undo conditions). If the bean owns other beans and uses bean slots properly (see bean(n)) the owned beans will be included in the delete set and will be undeleted with their owner.
pot dump
Returns a dump of information about all beans in the pot.
pot forget id
Private. Unregisters the bean with the given id when it is destroyed. This command is solely for the use of the bean(n) and should not be used by other clients.
pot get id
Returns the bean object with the given ID, throwing an error if there is no such bean in the pot.
pot has id
Returns 1 if the pot contains a bean with the given id, and 0 otherwise.
pot hasa cls id
Returns 1 if the pot contains a bean with the given id which is also a member of the given cls (or one of its subclasses) and 0 otherwise.
pot ids ?cls?
By default, returns a list of the IDs of all beans in the pot. If cls is given, returns a list of the IDs of the beans that are members of that class or its subclasses.
pot lastid
Returns the unique ID of the most recently created bean in the pot (which is, not coincidentally, the bean with the largest ID number).
pot markchanged
Sets the pot's changed flag. A bean instance should call this when it knows that its data has changed. Note that all bean(n) subcommands that modify the bean's data call this routine automatically.
pot new cls ?args...?

Creates an instance of the named bean(n) subclass, cls, passing any additional arguments to the subclass constructor, and returning the new object's name. The new bean is automatically assigned an ID and placed in the pot.

The bean's name will include the pot, an indicator of the bean's subclass, and the bean's ID, as an aid to debugging.

pot nextid
Returns the unique ID that will be assigned to the next bean to be created. Application code should generally not need to call this, though it might be useful in some circumstances. Note that setnextid can be used to control ID assignment.
pot reset
Deletes all beans from the pot resets the pot to its creation state. This command is primarily for use by the test suite, though it can also be called when creating a new scenario.
pot restore checkpoint ?-saved?
Triggers the restoration of a checkpoint stored in the -rdb; the checkpoint value is empty and ignored. By default, the changed flag is set; if -saved is given, the changed flag is cleared. (This is part of beanpot(n)'s saveable(i) interface.)
pot setnextid nid

Sets the bean ID, nid, to be returned the next time nextid is used to assign a bean ID. The nid must be greater than lastid to have any effect.

This command is for use in order(n) setredo scripts, to ensure that an order produces the same bean IDs on redo.

pot uncreate bean
This command is used by the undo code for bean(n) mutators that create new beans. This command verifies that the bean is the most recently created bean and destroys it
pot undelete deleteSet
Undoes a call to delete given the deleteSet returned by delete. Normal undo conditions apply; if any other changes have been made to beans since the deletion, they must be undone first.
pot validate id
Returns id unchanged if there is a bean with that ID in the pot, and throws INVALID otherwise.
pot valclass cls id
Returns id unchanged if there hasa bean with that class and ID in the pot, and throws INVALID otherwise.
pot view id ?view?
Returns a view dictionary for the bean with the given id.
