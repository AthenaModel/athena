DESCRIPTION

smartdomain(n) is a domain handler for ahttpd(n) that is used to handle dynamic application content. The application subclasses smartdomain(n); the subclass defines specific URLs and handler methods for them. The URLs can contain place-holder variables, which will be automatically matched and provided to the handlers.

TBD: Eventually we will rearchitect mydomain(n) to be a simple server that serves up smartdomain(n) content; then one smartdomain(n) instance can be used with both ahttpd(n) and mydomain(n). That work has not yet been done.

A smartdomain is associated with a particular server domain, i.e., with a subtree of the server's file system: e.g, "/my/subdomain" or "/mydomain". All URLs defined with the smartdomain will be within this subtree. In addition, the smartdomain defines the URL "domain/urlschema.html" which documents the accepted URLs within the domain.

Fallbacks

If a requested URL matches a directory within the domain, and that directory defines an "index.html" file, the server will be redirected to the "index.html" file. Otherwise, the content will be "not found".

COMMANDS

::projectlib::smartdomain is an oo::class. It provides the following commands.

::projectlib::smartdomain create name domain
Creates a new instance of smartdomain(n) with the given command name for the given domain. The domain must begin with "/".
::projectlib::smartdomain new domain
Creates a new instance of smartdomain(n) for the given domain and returns the instance's name. The domain must begin with "/".
smarturl cls suffix docstring body
This command is used outside of the definition of smartdomain(n) subclass cls to add URLs to the subclass. The suffix and docstring are defined as for url, below; and the body is the handler's method body. The handler method will be defined with one argument for each placeholder in the suffix, named as in the suffix.

INSTANCE COMMANDS

Instances of smartdomain(n) provide the following subcommands.

obj ahttpd
Installs the smartdomain for use with ahttpd(n), which should already be initialized.
obj mydomain name
Installs the smartdomain for use with the named mydomain(n). TBD: Not yet implemented.
obj domain
Returns the domain's domain, e.g., "/secenario".
obj redirect url
Redirects the requester to the given url by throwing HTTPD_REDIRECT.
obj request op suffix ?query?
Makes a direct request to the domain handler for the given suffix. The op may be GET or POST; on GET, the query will be parsed as a dictionary of query parms and values and passed to the handler. On POST it will not be. The full query string can always be retrieved using req.
obj url suffix docstring

Defines a new URL within the domain. The suffix is the URL's suffix within the domain, beginning with a "/" and ending with a file name and extension. For example, if the domain is "/mydomain" and the full URL will be "/mydomain/this/that/theother.html", then the suffix should be "/this/that/theother.html".

The file extension is required; it is used to determine the URL's content-type.

The suffix may contain place-holders for any path component but the filename, e.g., "/actor/{a}/index.html" or "/group/{g}/index.html". The place-holder is a variable that will be matched against the requested URL. If "/actor/fred/index.html" is requested, then "{a}" will match "fred". The place-holder must be the entire path component, and must be wrapped in "{" and "}" as shown.

The subclass must define a method with the same name as the suffix; this method will handle the URL. It must have one argument for each placeholder in the suffix. The handler may may:

  • Return data of a type consistent with the URL's file extension.
  • Throw NOTFOUND to indicate that the place-holder values match no available data (i.e., if there is no actor with the given name).
  • Call redirect to redirect to a different URL. Note that this causes the handler to throw HTTP_REDIRECT.

The docstring should describe the data returned by the URL in general terms, and document any place-holders.

obj urltree root docstring

TBD: This handler type was added to support the on-line help pages, but we might write a different domain handler for that purpose. In that case, this might go away.>

Defines a handler for a subtree of the domain. The root is the subtree's root within the domain, beginning with a "/". For example, if the domain is "/mydomain" and the full URL will be "/mydomain/this/that/...", then the root should be "/this/that".

The root may contain place-holders in any path component (as for url). Note that url and urltree handlers are matched in the order of definition; if the first registered handler is an urltree handler with root "/{mydir}" then it will match all URLs received by the domain and no other handler will ever be called.

The handler is a method with the same name as the root. It must have one argument for each place-holder in the root, giving the actual content from the requested URL, plus an argument for the remainder of the URL following the root. The handler may:

  • Return data of a type consistent with the URL's file extension.
  • Throw NOTFOUND to indicate that the place-holder values match no available data (i.e., if there is no actor with the given name).
  • Call redirect to redirect to a different URL. Note that this causes the handler to throw HTTP_REDIRECT.

The content-type of the returned data will be determined from the file extension of the last component of the requested URL; if there is no extension, then it will be "text/plain". TBD: This is not very good. If we retain this feature, we need a way to specify the default content-type for the subtree, and to set it explicitly for particular requests.

The docstring should describe the data returned by the URL in general terms, and document any place-holders.

COMPONENTS

smartdomain(n) defines the following component objects for use by URL handlers:

hb subcommand ?args...?
An instance of htmlbuffer(n) which can be used to generate HTML output.
qdict subcommand ?args...?
An instance of parmdict(n) which will contain any parsed query parameters and their values.
req parm

A command that returns data about the request being processed. The parm may be one of the following:

  • method: The HTTP method being used, GET or POST.
  • query: The full query string from the URL.
  • url: The full server-relative URL being handled, not including any query or fragment.
